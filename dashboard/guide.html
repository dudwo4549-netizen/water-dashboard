실시간 대시보드 데이터 연계

이제 AI 챗봇이 단순히 문서(RAG)만 읽는 것이 아닙니다. 대시보드 상의 실시간 수치 데이터를 문맥(Context)으로 함께 참조하여 현장 상황에 특화된 분석 자문을 제공합니다.

  • 분석 대상: 주차별 관리 현장 수, 전체 평균 실적률, 등급 상태(정상/집중관리/경고), 현장별 상세 미달 수량 등
  • 사용자 질의 예시: "현재 경고 상태인 현장들의 요약과 조치 방향을 알려줘."
🛡️

복원력(Resilience) 설계

네트워크 연결 불안정이나 구글 API의 일시적인 로드 급증 상황에서도 챗봇이 갑자기 멈추지 않고, 스스로 재시도하며 마지막 수단으로 명확한 조치법을 안내하도록 설계되었습니다.

  • 3회 지수 백오프 자동 재시도
  • 친절한 한국어 안내 메시지 표출

📊 챗봇의 실시간 데이터 분석 원리

대시보드 상에서 사용자가 주차(Week)를 바꾸거나 탭을 전환하면, 프론트엔드의 dashboardData 세션 값이 동적으로 갱신됩니다. 사용자가 챗봇에 메시지를 보낼 때, 현재 활성화된 화면의 모든 현장 요약 데이터가 프롬프트 서두에 결합되어 구글 Gemini API로 전송됩니다. 이를 통해 실시간 통계 분석과 지침서(RAG) 검색 근거가 결합된 초정밀 자문 답변이 탄생합니다.

Gemini API 서버 일시적 과부하 (503) 현상 안내

"Gemini API 서버가 현재 일시적인 과부하 상태(503)입니다. 잠시 후 다시 시도해 주십시오."

위 메시지는 챗봇 내부에 설계된 3회 자동 재시도(지수 백오프)가 모두 실패할 정도로 구글 AI 엔진에 트래픽이 폭증했을 때 발생합니다.

🛠️ 오류 발생 시 대처 가이드 (Action Item)

1단계: 1~2분 정도의 시간 여유를 두고 질문을 다시 전송해 봅니다. ▼
구글 Gemini API의 503(Service Unavailable) 및 429(Rate Limit) 오류는 일시적인 요청 급증 스파이크(Spike) 현상에 가깝습니다. 대부분 30초에서 2분 내에 서버 부하가 해소되므로, 다른 조작 없이 입력창에 질문을 다시 입력하여 전송(Send)하시면 곧바로 정상 답변을 받아보실 수 있습니다.
2단계: 브라우저 화면을 새로고침(F5) 합니다. ▼
네트워크 소켓이 지연되었거나 브라우저 캐시로 인해 세션 연결이 중단된 경우일 수 있습니다. F5 또는 Ctrl + F5를 눌러 페이지를 새로고침한 뒤 대시보드 화면 우측 하단의 챗봇 아이콘을 열어 다시 시도하십시오.
3단계: 우측 상단의 설정(⚙️) 메뉴에서 API Key 상태를 점검합니다. ▼
등록된 API Key가 비정상적이거나 만료된 상태여도 503 혹은 400 계열의 에러가 발생할 수 있습니다. 대시보드 우측 상단의 ⚙️ 모양 아이콘 버튼을 클릭하여 현재 등록된 Gemini API Key가 정상적인지 확인하고, 필요 시 구글 AI 스튜디오에서 새로운 API Key를 생성하여 다시 대시보드에 임시 등록해 보십시오.

GitHub Pages로 대시보드 공유하기

로컬 PC에서 실행 중인 통합 대시보드(dashboard 폴더)를 GitHub에 업로드하면, 서버를 상시 켜두지 않고도 무료 정적 웹 호스팅(GitHub Pages) 링크를 생성하여 팀원들과 실시간으로 공유할 수 있습니다.

🚀 깃허브 업로드 및 배포 절차 (5단계)

1단계: Local Git 저장소 초기화 및 불필요한 파일 제외

프로젝트 루트 디렉토리에서 터미널(PowerShell 또는 Bash)을 열고 아래 명령어를 순서대로 실행합니다. (백엔드 가상환경이나 불필요한 대형 압축파일은 .gitignore를 작성해 깃 대상에서 제외합니다.)

PowerShell / CMD (.gitignore 생성 및 git 초기화)
# 1. 깃 초기화 git init # 2. 불필요 파일 및 폴더 제외용 .gitignore 파일 생성 echo ".venv_nlm/" >> .gitignore echo "__pycache__/" >> .gitignore echo "*.zip" >> .gitignore echo ".agent" >> .gitignore # 3. 파일 스테이징 및 최초 커밋 git add . git commit -m "feat: RAG 대시보드 및 AI 챗봇 실시간 데이터 연계 완료"

2단계: GitHub에 새 리포지토리 생성 및 로컬 연결

개인 GitHub 계정에 로그인한 뒤, New Repository를 생성합니다. (리포지토리명 예: water-dashboard)
생성된 GitHub 주소를 복사하여 로컬 터미널에 연결하고 최초 푸시(Push)를 수행합니다.

GitHub 원격 저장소 연결 & 푸시
# 1. 브랜치명을 main으로 설정 git branch -M main # 2. 본인의 깃허브 저장소 주소를 연동 (username과 repo-name 수정 필요) git remote add origin https://github.com/사용자계정명/리포지토리명.git # 3. 원격 저장소로 코드 전송 git push -u origin main

3단계: GitHub Pages 기능 활성화

코드 푸시가 완료되면, 해당 리포지토리의 웹 페이지 상에서 다음과 같이 호스팅을 켭니다.

  1. GitHub 리포지토리 화면 우측 상단의 ⚙️ Settings 탭을 클릭합니다.
  2. 좌측 사이드바 메뉴에서 Code and automation 하위의 Pages 메뉴를 선택합니다.
  3. Build and deployment 섹션의 Source가 Deploy from a branch로 설정되어 있는지 확인합니다.
  4. Branch 항목에서 main 브랜치를 선택하고 폴더는 /root 또는 대시보드 구조에 따라 지정한 후 Save 버튼을 클릭합니다.

4단계: 배포 웹 링크로 접속

1~2분 정도 대기한 후 Settings > Pages 페이지를 새로고침하면 상단에 "Your site is live at..."라는 안내와 함께 고유 URL 링크가 나타납니다.

🔗 공개 도메인 주소 형태: https://사용자계정명.github.io/리포지토리명/dashboard/index.html

※ 이 링크를 사용하면 모바일 폰이나 회사 외부 PC에서도 크롬 브라우저를 열어 우측 설정에서 API Key를 입력한 뒤, 현장 실시간 유수율 상태 및 AI 챗봇을 즉각 테스트해 볼 수 있습니다.
